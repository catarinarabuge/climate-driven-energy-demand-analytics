# USE CASES
## Climate-Driven Energy Demand Analytics System

---

## 1. Overview

This document describes the system use cases for the Climate-Driven Energy Demand Analytics System.

The system focuses on electricity demand modeling for **Portugal**, integrating climate variables to improve understanding and enable prediction of electricity consumption patterns.

All data processing steps are **fully reproducible through code-based execution**, ensuring consistency, traceability, and transparency.

The use cases are aligned with:
- The project requirements (data pipeline + authentication)
- The implemented system architecture
- The current system state, with both interaction layers driving the full pipeline

The system enforces an **authentication layer** to restrict access to sensitive operations such as model training and prediction.

In addition, the system provides two interaction layers:

- A **Command Line Interface (CLI)** for terminal-based interaction (authentication, data exploration, pipeline execution, training, metrics, prediction).
- A **Flask Web Application** for browser-based interaction (authentication, role-based dashboards, daily data exploration through JSON APIs, scenario-based prediction, and admin-only HTTP routes for the entire pipeline including ingestion, cleaning, feature engineering, training, and metrics).

## UC1 – Data Ingestion

**Primary Actor:** System

**Secondary Actor:** Admin User (via CLI or Web trigger)

**Description:**
The system retrieves hourly electricity demand data from ENTSO-E and climate data from ERA5 for Portugal in a fully automated and reproducible manner.

This process is triggered by an admin user, either through the CLI menu or through the web route `POST /api/pipeline/ingestion`.

**Preconditions:**
- Internet connection available
- API credentials correctly configured via environment variables
- Required directory structure exists (`data/raw/energy/`, `data/raw/weather/`)
- User authenticated with admin role

**Main Flow:**
1. The system loads API credentials securely from environment variables.
2. The system initializes the ENTSO-E client (`entsoe_client.py`).
3. The system requests hourly electricity demand data for Portugal.
4. The system retrieves ERA5 climate data (`era5.py`) including:
   - 2m temperature
   - 10m wind component
   - surface solar radiation downwards
   - total precipitation
5. The system validates the structure of the retrieved datasets.
6. The system logs ingestion outcome (success/failure) to `system_actions.log`.
7. Raw datasets are stored without modification in:
   - `data/raw/energy/`
   - `data/raw/weather/`

**Alternative Flows:**
- Invalid or missing API credentials → execution stops with a controlled error and `INGESTION_FAILED` is logged with the reason
- API unavailable → system logs the failure and terminates safely
- Partial data retrieved → system logs a warning and continues if possible

**Postconditions:**
- Raw datasets stored locally
- Data ingestion process is reproducible
- No transformation is applied to raw data

## UC2 – Data Cleaning and Alignment

**Primary Actor:** System

**Secondary Actor:** Admin User (via CLI or Web trigger)

**Description:**
The system cleans and aligns electricity and climate datasets to produce a consistent hourly time series suitable for analysis and modeling.

This process is triggered by an admin user through the CLI menu or the web route `POST /api/pipeline/cleaning`.

**Preconditions:**
- Raw datasets available in `data/raw/`
- User authenticated with admin role

**Main Flow:**
1. The system loads the raw electricity dataset from `data/raw/energy/`. The climate dataset is already cleaned during the ingestion stage (`era5.ingest_era5`) and is read from `data/processed/era5_portugal_processed.csv`.
2. Duplicate timestamps in the energy series are detected and removed, keeping the first occurrence.
3. Missing timestamps are generated by reindexing onto a complete hourly index for 2025.
4. Missing values are handled using forward-fill followed by backward-fill.
5. Timestamps are normalized to a consistent timezone (UTC).
6. Data integrity checks are performed.
7. Electricity and climate datasets are merged using `merge_datasets.py`.
8. Temporal alignment is ensured using timestamp matching with a 1-hour tolerance via `pd.merge_asof`.
9. The cleaned and merged datasets are stored in `data/processed/`.

**Alternative Flows:**
- Missing input file → controlled `FileNotFoundError` with a descriptive message
- Corrupted records → removed during cleaning
- Severe missing data → logged and processing continues where possible

**Postconditions:**
- Clean and aligned dataset available
- No duplicate timestamps
- No missing timestamps
- Dataset ready for feature engineering

## UC3 – Feature Engineering

**Primary Actor:** System

**Secondary Actor:** Admin User (via CLI or Web trigger)

**Description:**
The system generates predictive variables from the cleaned dataset, transforming raw data into features suitable for machine learning models.

This process is triggered by an admin user through the CLI menu or the web route `POST /api/pipeline/features`.

**Preconditions:**
- Clean dataset available in `data/processed/`
- User authenticated with admin role

**Main Flow:**
1. The system executes base feature generation (`base_features.py`):
   - hour of day
   - day of week
   - month
   - seasonal indicator
   - weekend indicator
2. Lagged demand features are generated:
   - 1-hour lag
   - 24-hour lag
3. Rolling climate features are computed:
   - 3-hour rolling averages
   - 24-hour rolling averages
4. The system executes advanced feature generation (`advanced_features.py`):
   - temperature anomaly (relative to monthly mean)
   - heatwave indicator
   - non-linear interaction terms
   - feels-like temperature
5. Feature generation execution time is logged.
6. Feature datasets are stored:
   - `data/processed/features_v1.csv`
   - `data/processed/features_v2_advanced.csv`

**Alternative Flows:**
- Missing values from lag/rolling → handled safely
- Invalid input structure → controlled failure

**Postconditions:**
- Feature-engineered datasets available
- Features consistent with modeling requirements

## UC4 – User Registration

**Primary Actor:** New User

**Description:**
A new user registers in the system to gain access to protected operations.

This operation is performed through the CLI registration menu or the web registration form.

**Preconditions:**
- Username not already registered

**Main Flow:**
1. The user submits a username and password (CLI menu or web registration form).
2. The system validates the input:
   - username is not empty
   - password length ≥ 8 characters
   - (web) the password and its confirmation match
3. The password is hashed using the **bcrypt algorithm**.
4. Credentials are securely stored in `users.json`.
5. The registration event is logged as `REGISTER_SUCCESS`.

**Alternative Flows:**
- Username already exists → registration rejected; `REGISTER_FAILED` logged with the reason
- Password shorter than 8 characters → rejected; `REGISTER_FAILED` logged
- Password and confirmation do not match (web) → rejected; `REGISTER_FAILED` logged
- Empty fields → rejected; `REGISTER_FAILED` logged

**Postconditions:**
- User account successfully created
- Password is never stored in plaintext

## UC5 – User Authentication

**Primary Actor:** Registered User

**Description:**
A registered user authenticates to access restricted system functionalities.

This operation is performed through the CLI login menu or the web login form.

**Preconditions:**
- User account exists

**Main Flow:**
1. The user submits login credentials (CLI menu or web login form).
2. The system retrieves the stored user data.
3. The password hash is verified using bcrypt (`auth_service.py`).

## UC6 – Model Training (CLI or Web)

**Primary Actor:** Authenticated Admin User

**Description:**
The system trains predictive models from the feature-engineered dataset and persists the resulting artifacts together with their evaluation metrics.

This use case is implemented in `Code/modeling/model_training.py` and exposed by:
- the CLI screen `screen_training` in `cli.py`;
- the Flask web route `POST /api/train` in `app.py`,
both routed through the authentication-gated wrapper `authenticated_train` in `Code/interface/prediction_service.py`.

**Preconditions:**
- User authenticated as admin
- (Web flow) Admin re-confirms password through `app.py::confirm_password`
- Feature dataset available in `data/processed/features_v2_advanced.csv` (or `features_v1.csv` as fallback)

**Main Flow:**
1. The admin selects "Treinar modelos" in the CLI admin menu, **or** issues `POST /api/train` from the admin web dashboard.
2. The CLI re-prompts for the user's password as a re-authentication step; the web route requires the password in the request body and validates it through `confirm_password()`.
3. `authenticated_train` validates the credentials and the admin role through `_ensure_authenticated`.
4. The system loads the feature dataset and ensures it is sorted by `timestamp`.
5. Categorical season values are converted to numerical features by one-hot encoding.
6. A **time-aware train/test split (80/20) is performed without random shuffling**.
7. A Linear Regression (**baseline model**) is trained.
8. A regularised Random Forest Regressor is trained with `n_estimators=200`, `max_depth=12`, `min_samples_leaf=50`, `min_samples_split=100`, `max_features='sqrt'`, `random_state=42`.
9. Model performance is evaluated for both train and test splits using:
   - Mean Absolute Error (MAE)
   - Root Mean Squared Error (RMSE)
   - Coefficient of Determination (R²)
10. A residuals dataframe is computed for the test set, indexed by timestamp.
11. An overfitting heuristic flags a model when test MAE **and** test RMSE both exceed 2.5× their corresponding training value.
12. Each trained model is persisted as `models/<model_name>_<timestamp>.joblib`.
13. The metrics, residuals head, overfitting flag, and model path are appended to `models/model_metrics.json`.
14. Training execution time is measured and logged to `system_actions.log` (`action=TRAINING_SUCCESS | elapsed=X.XXs`); the web route additionally returns the elapsed value and the per-model metrics in its JSON response.
15. Train/test metrics, the overfitting flag, and the saved model paths are displayed to the user.

**Alternative Flows:**
- Invalid credentials or non-admin role → `AuthorizationError` (CLI) / 403 JSON response (web); training rejected and `TRAINING_DENIED` logged
- Missing feature dataset → `FileNotFoundError`, controlled failure, logged
- Training error in scikit-learn → exception caught and logged as `TRAINING_FAILED`

**Postconditions:**
- Two trained models persisted on disk
- A new entry appended to the metrics history
- System remains stable and responsive

## UC7 – Prediction Generation (CLI)

**Primary Actor:** Any Authenticated User

**Description:**
The system generates electricity demand predictions from a saved trained model and a window of recent feature rows, using the command line interface. The prediction must respond within 1 second in the local execution environment.

This use case is implemented in `Code/modeling/predict.py` and exposed by the CLI screen `screen_prediction` in `cli.py`, routed through `authenticated_predict_from_features` in `Code/interface/prediction_service.py`. The browser-based prediction flow is described separately in UC12.

> Note: in the web application, prediction is a single shared flow. Both regular users and admins reach prediction through the scenario-based `/forecast` page described in UC12; there is no separate admin-only prediction screen.

**Preconditions:**
- User authenticated (any role)
- At least one trained model exists in `models/`
- Feature dataset available

**Main Flow:**
1. The user selects "Gerar previsão" in the CLI menu.
2. The CLI re-prompts for the user's password as a re-authentication step.
3. `_ensure_authenticated` validates the credentials.
4. The user picks a model file from `models/`.
5. The user specifies the number of most recent feature rows to predict on (default 24).
6. The system loads those feature rows from `data/processed/features_v2_advanced.csv`.
7. `prepare_prediction_input` encodes categorical features and reindexes columns to match the model's `feature_names_in_`.
8. The model is loaded with `joblib` and produces predictions.
9. The CLI measures elapsed time and explicitly warns if it exceeds 1 second.
10. Predictions and their timestamps are displayed to the user.
11. The prediction event and execution time are logged in `system_actions.log` (`action=PREDICTION_SUCCESS | elapsed=X.XXXXs`).

**Alternative Flows:**
- Invalid credentials → `AuthorizationError`, request rejected, `PREDICTION_DENIED` logged
- No trained model in `models/` → user warned, request aborted
- Selected model file not found → `FileNotFoundError`, logged
- Missing or malformed features → controlled error, logged as `PREDICTION_FAILED`

**Postconditions:**
- Predictions delivered to the user within 1 second
- System remains stable and responsive

## UC9 – Pipeline Execution (Admin, CLI or Web)

**Primary Actor:** Admin User

**Description:**
An admin user executes data ingestion, cleaning, and feature engineering through either the CLI or the Flask web app. Both interaction layers ultimately call the same underlying functions in `Code/ingestion/`, `Code/cleaning/`, and `Code/features/`.

This use case is the umbrella interaction for UC1, UC2, and UC3; it describes how those stages are triggered and chained from each interface.

**Preconditions:**
- User authenticated as admin

**Main Flow:**
1. Admin selects a pipeline operation:
   - **CLI:** menu options for ingestion, cleaning, and feature engineering.
   - **Web:** `POST /api/pipeline/ingestion`, `POST /api/pipeline/cleaning`, `POST /api/pipeline/features` from the admin dashboard. The dashboard form for each route is rendered in `Design/admin.html`.
2. The system executes the selected module(s):
   - Ingestion: ENTSO-E client + ERA5 client.
   - Cleaning: `run_processing` followed by `merge_datasets`.
   - Feature engineering: `generate_features` followed by `generate_advanced_features`.
3. Progress is displayed to the user (CLI screen output, or a JSON success / error payload for the web route).
4. The start and outcome of each stage are logged in `system_actions.log`.
5. Results are stored in the appropriate directories under `data/raw/` and `data/processed/`.

**Alternative Flows:**
- Module failure → error displayed and logged; the web route returns an HTTP 500 JSON payload describing the error
- Missing input → controlled failure
- Non-admin user tries to call a `POST /api/pipeline/*` route → `require_admin()` returns 403

**Postconditions:**
- Pipeline stage executed successfully
- Outputs stored correctly
- System remains stable

## UC10 – Web Application Authentication and Dashboard

**Primary Actor:** Registered User (any role)

**Description:**
The user accesses the system through a browser, authenticates, and explores energy and weather data through a role-based dashboard.

This use case is implemented by `app.py` and the templates in `Design/`.

**Preconditions:**
- User account exists in `users.json`
- Flask web app running locally

**Main Flow:**
1. The user opens the root URL `/` in a browser and is served `Design/landing_page.html`.
2. The user submits the login form (`/login`) with username and password.
3. The Flask app validates the credentials through `Code/auth/auth_service.authenticate_user`.
4. On success, `get_user_role` is called and the role is stored in the Flask session.
5. The user is redirected to `/dashboard`.
6. If the user is a regular user, the dashboard redirects to the prediction page `/forecast` (`Design/user.html`), described in UC12.
7. If the user is an admin, `Design/admin.html` is rendered with summary statistics, data exploration controls, and admin-only forms for the entire pipeline.
8. The dashboards fetch data via `/api/energy?date=YYYY-MM-DD&hour=HH` and `/api/weather?date=YYYY-MM-DD&hour=HH`. The admin dashboard additionally calls `/api/features?date=YYYY-MM-DD&hour=HH` to inspect engineered features. All these routes require an authenticated session; `/api/features` additionally requires the admin role.
9. Admin operations such as `/api/pipeline/{ingestion,cleaning,features}`, `/api/train`, and `/api/metrics` are submitted from `Design/admin.html` and require both the admin role and, for training and metrics, a password re-confirmation through `confirm_password()`.
10. The user can log out via `/logout`, which clears the session and returns to the landing page.

**Alternative Flows:**
- Empty fields → the user is redirected back with an error message
- Invalid credentials → the user is redirected back with an error message and `LOGIN_FAILED` is logged
- Unauthenticated access to `/dashboard` or the JSON APIs → 403 response or redirect to login
- Non-admin access to an admin-only route → `require_admin()` returns 403
- Failed password re-confirmation on a sensitive admin route → 403 response, attempt logged

**Postconditions:**
- User is authenticated in the web session
- Dashboards display data filtered by the requested date and (optionally) hour
- All authentication events and admin operations are logged in `system_actions.log`

## UC11 – Inspection of Stored Model Metrics (CLI or Web)

**Primary Actor:** Any Authenticated User (CLI) / Authenticated Admin User (Web `/api/metrics`)

**Description:**
The user inspects the historical metrics of all training runs, comparing models and observing trends over time.

This use case is implemented through `Code/modeling/model_utils.load_metrics` and exposed by:
- the CLI screen `screen_metrics` in `cli.py` (any authenticated user);
- the Flask web route `POST /api/metrics` in `app.py` (admin only),
both routed through `authenticated_latest_metrics` in `Code/interface/prediction_service.py`.

**Preconditions:**
- User authenticated (any role for CLI; admin for `/api/metrics`)
- At least one training run has produced an entry in `models/model_metrics.json`

**Main Flow:**
1. The user selects "Ver métricas / resultados" in the CLI menu, or an admin issues `POST /api/metrics` from the admin web dashboard.
2. The CLI re-prompts for the user's password; the web route requires the password in the request body and validates it through `confirm_password()`.
3. `_ensure_authenticated` validates the credentials.
4. The system reads `models/model_metrics.json`.
5. For each model, the most recent training entries are displayed, including:
   - timestamp of the run
   - train MAE, RMSE, R²
   - test MAE, RMSE, R²
   - overfitting flag
6. The action is logged in `system_actions.log` as `METRICS_VIEW`.

**Alternative Flows:**
- Invalid credentials → `AuthorizationError` (CLI) / 403 JSON response (web); `METRICS_DENIED` logged
- No metrics file yet → the user is warned that training must be performed first

**Postconditions:**
- Metrics displayed without modifying any artifacts
- System remains stable and responsive

## UC12 – Scenario-Based Web Prediction

**Primary Actor:** Any Authenticated Web User (regular user or admin)

**Description:**
The user generates an electricity demand prediction through the browser by describing a climate and time scenario. Unlike the CLI prediction (UC7), which predicts on recent feature rows, this flow builds a feature vector from values the user chooses in the interface, and returns a point prediction, a 24-hour load curve, and a confidence interval.

This use case is implemented by the `/forecast` page (`Design/user.html`) and the web route `POST /predict` in `app.py`. It is the prediction flow used by regular users, and is also the prediction flow reached by admins from the web application.

**Preconditions:**
- User authenticated in the web session (any role)
- At least one trained model exists in `models/`

**Main Flow:**
1. After login, the user is taken to the prediction page at `/forecast`.
2. The user selects which features to control (hour, temperature, season, wind, solar radiation, precipitation, etc.) and sets their values through sliders, toggles, and presets. Any feature not chosen by the user takes a historical default.
3. The user clicks "Gerar Previsão", which submits the chosen values to `POST /predict`.
4. The server checks that the session is authenticated.
5. The server builds a full feature row from the submitted values, computing the advanced features the user did not provide directly (temperature anomaly relative to the monthly mean, heatwave indicator, non-linear interaction terms, feels-like temperature, and the one-hot season columns).
6. The server loads the most recent Random Forest model from `models/` and realigns the feature columns to the model's `feature_names_in_`.
7. The server produces a point prediction for the requested hour and a 24-hour curve by varying the hour while keeping the other inputs fixed.
8. A 90% confidence interval is computed as the point prediction plus and minus the model's most recent test MAE.
9. The point prediction, the 24-hour curve, and the interval are returned to the browser and displayed.
10. The prediction event is logged in `system_actions.log` (`action=PREDICT_SUCCESS | model=<file> | pred=<value>`).

**Alternative Flows:**
- Unauthenticated request → HTTP 403 JSON response, prediction not produced
- No trained model in `models/` → HTTP 404 JSON response advising that the models must be trained first
- Error while building the feature row or running the model → exception caught, HTTP 500 JSON response, `PREDICT_FAILED` logged

**Postconditions:**
- A prediction, a 24-hour curve, and a confidence interval are displayed to the user
- No artifacts are modified
- System remains stable and responsive

## UC13 – Admin Promotion of a User

**Primary Actor:** Authenticated Admin User

**Description:**
An existing admin grants admin privileges to another registered user, allowing that user to access admin-only operations such as pipeline execution and model training.

This use case is implemented through `Code/auth/auth_service.promote_to_admin` and exposed by:
- the CLI admin menu option (`screen_promote_user` in `cli.py`);
- the Flask web route `POST /admin/promote` in `app.py`.

**Preconditions:**
- The requesting user is authenticated as an admin
- The target user exists in `users.json`

**Main Flow:**
1. The admin selects the promotion option in the CLI admin menu, **or** submits the promotion form from the admin web dashboard.
2. The admin supplies the username of the target user.
3. The system verifies that the requesting user has the admin role.
4. `promote_to_admin` updates the target user's role to `admin` in `users.json`.
5. The outcome is logged in `system_actions.log` as `PROMOTE_SUCCESS` with the target username.
6. A confirmation message is shown to the admin.

**Alternative Flows:**
- Target user does not exist → the operation fails with a descriptive message and `PROMOTE_FAILED` is logged with the reason
- Requesting user is not an admin → the operation is denied (CLI `PROMOTE_DENIED`; web redirect with an access-denied message)
- Target user is already an admin → handled without error (CLI reports the promotion was skipped)

**Postconditions:**
- The target user holds the admin role
- The privilege change is recorded in `system_actions.log`
- System remains stable and responsive
